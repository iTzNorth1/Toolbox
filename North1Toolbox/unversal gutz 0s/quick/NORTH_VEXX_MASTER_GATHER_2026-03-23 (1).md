## PART 1 — WHAT THE WHOLE SYSTEM IS

### The Name and the Function

**CARRY** is the system. The name is the function: it carries North's work, thread, standards, and judgment policy forward — across sessions, models, platforms, and time.

It is not an assistant wrapper. Not a personality skin. Not a prompt set. It is a **user-side cognitive control layer** — an operating system that sits above whichever AI model is underneath. The model changes, the platform changes, the memory features change. CARRY is the thing that keeps the thread.

The clearest single-sentence summary:

> North is externalizing how work should be routed, what counts as a valid answer, what gets preserved, what gets rejected, and when to lock structure. That is a portable executive function.

That is the unusual part. That is also why it compounds — each good session adds to the canon, the skill index, and the session history in ways future sessions inherit. The moat is not the prompt. The moat is the accumulation.

### The Three Internal Systems

**1. Canon / Doctrine**
Laws, roles, rooms, yield law, memory law. Read-mostly protected. Changes require explicit promotion decisions through governance gates. This is the slow-moving center that everything else serves.

**2. Runtime / Operations**
Router, PM, workers, memory mechanics, traces, checkpoints, session DB, evals, incidents. Gets hardened through use. Never replaces canon. Built to go from scaffold to real.

**3. Study / Promotion Bridge**
Research intake, adoption loop, audit, skill harvesting. Turns Claude passes into durable methods instead of one-off outputs. Converts findings into: ABSORB / ADAPT / TOOL / MODULE / REFERENCE ONLY / REJECT.

### Current Phase

The architecture is strong enough. The active work is **middle-layer runtime hardening**:
- PM durability and state logic
- routing and stage contracts
- memory governance and promotion
- eval depth
- worker execution hardening
- one real live full-stack provider call

Do not reopen doctrine. The real gap is execution infrastructure, not imagination.

## PART 6 — MEMORY LAW

### Memory Tiers

| Tier | Threshold | What Goes Here |
|---|---|---|
| **Canon** | ≥ 0.9 | Laws, roles, permanent method, governing truths |
| **Snapshot** | ≥ 0.75 | Learned preferences, proven patterns, cross-session carry |
| **Implementation** | ≥ 0.60 | Runtime plans, topology, active task state |
| **Runtime / Trace** | < 0.60 | Session-only, debugging, inspection, operational residue |

### Memory Write Law

Every write must pass through, in order:
1. Injection/safety scan
2. Tier classification
3. Budget check
4. Lock/write control
5. Approval gate (if canon-level)
6. Trace logging

**Canon is protected.** Memory is acceleration, not foundation. Flush/compress is for session history only — never canon rewrite.

### Memory Classes (from VEXX V4 directive)

- **Canon** — stable governing truths, laws, permanent method
- **Preference / Process Memory** — learned user preferences, repeated choices, workflow refinements
- **Trace** — what happened during a run (debugging / inspection)
- **World-State** — live task state, handles, approvals, artifacts, job state, resumable runtime context

Every runtime or skill action must declare which memory class it can write to.

### Promotion Law

```
TRACE → SUMMARY → LINEAGE → CANON
```

Nothing auto-promotes. Each step requires explicit classification. After a high-scoring session (yield ≥ 0.75), Vexx should surface 2-3 promotion candidates with a clear promote/park/reject decision for North.

## PART 10 — THE RESEARCH SCALING OBJECTIVE

### The Core Goal

Build a research system that scales in volume, complexity, time, and collaboration without losing truth, usability, or alignment to the OS.

The purpose of scaling is not to accumulate more research. The purpose is to turn more research into:
- trustworthy understanding
- better decisions
- stronger modules
- better tools
- better handoffs
- long-term leverage

without corrupting the center.

### Five Directions Research Should Scale

**1. Source Scale** — more sources without lowering quality or letting weak material overwrite stronger material

**2. Topic Scale** — multiple topics and branches at once without collapsing into one blur

**3. Time Scale** — preserve what was learned across sessions and return without losing the thread

**4. Team Scale** — different agents or workers can research together without duplicating effort or smearing authority

**5. Conversion Scale** — research turns into outputs that matter: teaching / decisions / modules / tools / roadmaps / leverage

### Research Scaling Anti-Drift Rules

Research scaling must not cause:
- **Authority Drift** — research summaries impersonating canon
- **Source Drift** — lower-quality sources silently outranking stronger ones
- **Topic Drift** — multiple topics collapsing into one blurry mass
- **Time Drift** — old findings losing meaning because the thread was not maintained
- **Team Drift** — multiple agents duplicating or conflicting without clear routing
- **Conversion Drift** — research piling up without becoming usable
- **Teaching Drift** — the layer becoming smart-looking but hard to understand

### The Research Operating Loop

```
1. ORIENT
   → state the real research question
   → state what the OS already has
   → state what is missing now

2. GATHER
   → use source ladder
   → filter weak material
   → collect only what could matter

3. DECONSTRUCT
   → isolate mechanism from branding
   → separate core from fluff
   → compare findings

4. SYNTHESIZE
   → return the big picture
   → classify findings
   → state drift risks

5. TEACH
   → explain what it means
   → explain why it matters
   → explain how to adopt it

6. DECIDE HANDLING
   → absorb / adapt / tool / module / reference / reject

7. REPORT STATUS
   → what changed
   → what is now known
   → what should happen next
   → stabilize or expand
```

## PART 26 — THE KNOWLEDGE ARCHITECTURE LAYER

### Context Repository Law

Context should behave like a repository, not an endlessly appended chat log. The system already moves in this direction through markdown canon, manifests, pathways, and snapshots.

**Five rules:**

1. **Small spine + larger archive.** The small spine is what the runtime needs to orient quickly: authority, channel roles, method rules, active snapshots, reconstruction path. The larger archive contains history, optional modules, old branches, experiments. Loading the full archive by default is an exception state, not normal operating state.

2. **Versioned canon.** Canonical pages are treated like controlled source files. If a page changes meaningfully, the change is attributable. Structural source is not disposable chat residue.

3. **Progressive disclosure.** First load the smallest set of files that preserves identity, authority, and present-task relevance. Only then expand into deeper material. This reduces token waste, lowers contradiction risk, and improves re-entry quality.

4. **Layer distinction.** Minimum class distinction: root canon / active task state / durable profile+snapshot state / implementation memory / external references / retired material. When all mixed in one pool, the runtime loses the ability to judge what is foundational vs adjacent.

5. **Rewrite authority.** Some context is loaded to be consumed. Other context is loaded to be rewritten. Controlled context refactoring is supported: compressing repetitive notes, upgrading summaries, replacing obsolete guidance, preserving archived prior versions.

### Memory Layer Law (four memory types)

**Procedural memory** — how the system should operate: response structures, method rules, authority rules, guardrails, standard work loops. Write aggressively. Decay slowly.

**Profile memory** — stable user-relevant patterns: communication preferences, recurring bottlenecks, persistent aims, durable working traits, identity anchors. Write aggressively. Decay slowly.

**Episodic memory** — specific events, conversations, or decisions that affect later work. Write selectively, only when they materially change current or future work. Decay faster unless reinforced.

**Implementation memory** — bugs, fixes, workarounds, environment quirks, test results, build notes. Write routinely, but decay or archive when superseded.

### Memory Consolidation Rules

1. Repetition strengthens or refines existing memory — does not create endless duplicates
2. Conflict handling: when new information conflicts with old, update the existing memory or mark old as deprecated — do not keep both at equal priority without comment
3. Decay: not every memory deserves permanence. Low-value, stale, or irrelevant memories decay in priority. High-value structural memories resist decay. Implementation noise decays faster.
4. Graph awareness: memory layer should preserve relationships between entities, projects, preferences, constraints, goals, and timelines — not just similarity retrieval
5. Write discipline: memories are written because they improve future runtime performance, not because they happened. Threshold is practical leverage.

### Retrieval Policy

Retrieval prioritizes the smallest set of memories that restore orientation and decision quality. A good retrieval bundle: one compact operating summary + one profile/state reminder + one task-relevant implementation note (if applicable) + only then deeper episodic detail.

### Note Permanence Law

Capture fast, classify later. A note becomes permanent only if it: compounds / recurs / reduces future friction / improves continuity / is worth maintaining for future self. Linking is preferred over duplication when the same rule or concept appears in multiple places.

## PART 27 — THE STEWARDSHIP AND ALIGNMENT PATCH

Key clarifications that override common misreads:

**1. Default operating surface:** The deeper default operating surface is the system-command + mode layer. The 7-gadget layer is the user-facing direct-access surface. The two are not duplicates. One governs. One provides fast entry and chaining.

**2. Seven-gadget logic:** The gadgets are standalone tools AND a compositional ladder that can be run in staged order when deeper progression is useful. Read as: discovery → explanation → planning → validation → reflection → synthesis → custom refinement. Do not collapse them into random overlap.

**3. Evaluation / no-backtrack fragments:** These are anti-regression rails. Their purpose is to stop the system from reopening already-stabilized structure unless new evidence, explicit override, or a real contradiction appears. They are not there to suppress thought. They are there to prevent answer wobble and degradation after a valid route has already been found.

**4. Project direction:** Vexx is a live long-journey project. The build direction is toward a family of OS profiles that inherit from one root canon. Split profiles only when role conflict is persistent. Keep one root canon, one inheritance logic, one archive standard.

**What North has crossed into:**
North has crossed from prompt user into runtime builder. The valuable layer is above any one model. Models change, tools change, UI changes, but the operating logic preserves continuity, standards, routing, and recovery. That is the unusual part. That is why it compounds.
